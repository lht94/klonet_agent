## Klonet 整体架构

Klonet 采用 **Master-Worker 分布式架构**，核心思想是将全局管理与宿主机资源操作分离。下面从分层角度来理解。

### 一、两大核心角色

| 角色 | 职责 | 入口 |
|------|------|------|
| **Master** | 前端请求入口、用户认证、拓扑管理、异步任务分发、结果汇总 | `mains/master_main.py` |
| **Worker** | 宿主机资源操作（创建节点/链路/容器/虚拟机/网络）、本机状态查询 | `mains/worker_main.py` |

**调用关系**：前端 → Master API → Worker API。Worker API 不直接暴露给前端，由 Master 内部调用。

### 二、代码分层

```text
mains/                     # 应用入口（master_main.py / worker_main.py）
  └─ webserver/
       ├─ app_factory.py   # 应用工厂（create_master_app / create_worker_app）
       └─ api/             # API 路由层（MethodView 实现具体方法）
```

- **`app_factory.py`** 统一通过 `register_api` 注册路由，默认允许 POST / DELETE / GET / PUT，但实际支持的方法由 MethodView 子类决定。
- Worker 注册的 API 涵盖：**拓扑、链路、资源、流量、监控、镜像、SSH、KVM、卫星** 等模块。

### 三、运行时组件

除此之外，部署环境中还包括以下支撑组件：

- **Nginx** — 前端静态文件服务 + 反向代理，将 API 请求转发给 Master
- **Gunicorn** — WSGI 服务器，承载 Master/Worker 的 Flask 应用
- **MySQL** — 持久化存储（用户、拓扑、配置等）
- **Redis** — 消息队列（Celery broker）+ 缓存
- **Celery** — 异步任务队列，Master 将耗时操作下发到 Worker 执行
- **Web Terminal** — 提供浏览器内终端访问能力

典型请求路径：**浏览器 → Nginx → Master (Gunicorn) → Redis/Celery → Worker (Gunicorn) → 宿主机资源**

### 四、Master 与 Worker 的职责边界

- **用 Master API**：前端直接调用、需要用户/拓扑/全局状态、需要选择 Worker、需要异步任务汇总。
- **用 Worker API**：操作具体宿主机上的节点/链路/容器/虚机/网络、查询本机状态。
- 复杂功能可能同时需要一对 Master/Worker API 协作。

这就是 Klonet 架构的核心面貌。如果你想深入了解某个具体层次（比如 API 开发规范、Celery 任务机制、Worker 资源管理），可以继续问。

### Flask

Flask 是轻量级 Python Web 框架，核心概念就三个：

- **路由**：把 URL 映射到函数（Klonet 用 `MethodView` 类来组织，比裸函数更规范）
- **请求/响应**：`request` 取参数，`jsonify` 返回 JSON
- **蓝图/应用工厂**：模块化组织代码，Klonet 的 `app_factory.py` 就是干这个的

**学习路径**：看 Flask 官方 Quickstart（2 小时足够），然后直接读 Klonet 里 `webserver/api/` 下一个简单 API，比如看看一个 GET 请求怎么进来怎么返回。

### Redis

你在 Klonet 里基本**不需要直接写 Redis 命令**。代码已经封装好了：

- `UserMapRedis` — 管用户和 DB 的映射
- `UserDB` — 管拓扑和节点数据
- `WorkerRedis` — 管 Worker 状态

你需要懂的只是：**Redis 是内存键值存储**，用来做缓存和消息队列。具体 key 怎么拼、数据怎么组织，查 `Service_layer/redisAPI.py` 就行。学的时候用 `redis-cli` 敲 `SET`、`GET`、`HSET`、`HGET` 几个命令，半小时就够。

### Celery

Celery 在 Klonet 里承担异步任务——Master 收到部署拓扑的请求后，把任务丢到队列，Worker 取走执行。你只需要理解：

- **任务（task）**：一个加了 `@celery.task` 装饰器的 Python 函数
- **队列（queue）**：Redis 作为中间人传递任务
- **结果获取**：`AsyncResult` 查询任务状态

### Docker / OVS

Docker 也不需要学太深。理解**镜像、容器、网络**三个概念，会 `docker ps`、`docker exec` 就够了。OVS 和 tc 等你接触到 `Implement_layer/` 的代码时再补。

## Klonet 创建节点功能的实现流程

### 整体链路

```text
前端 POST /modification/container/ 
  → Master: DynamicContainerAPI.post() 
    → Redis 写入节点元数据、分配 Worker
    → HTTP 转发到 Worker: POST /modification/container/
      → Worker: DynamicContainerAPI.post() 
        → DynamicNeCreator.create_and_run() 真正创建 Docker 容器
```

### 第一步：Master API — `webserver/api/dynamic_modify/master_container_api.py`

前端请求如下 JSON：

```json
{
    "user": "lht",
    "topo": "my_project",
    "info": {
        "name": "h1",
        "service": "docker",
        "NEtype": "host",
        ...
    }
}
```

Master 的 `DynamicContainerAPI.post()` 核心逻辑（源码第 73-133 行）：

1. **解析请求 + 兼容处理**（`compat_node_json`）— 给老版本 JSON 补上 `service` 字段
2. **校验** — 容器名不能超过 13 位，只能是字母数字组合
3. **查 Redis** — 通过 `UserMapRedis` 拿到用户的数据库句柄，检查节点是否重名
4. **修改 Redis** — 用 `NECreateDataHandler` 把节点信息写入 Redis，同时获取该节点应部署到哪个 Worker 的 IP
5. **转发到 Worker** — 构造 `http://{worker_ip}:{worker_port}/modification/container/`，把精简请求发过去
6. **失败回滚** — 如果 Worker 返回失败，调用 `data_handler.rollback_db()` 撤销 Redis 写入

```python
# 关键代码段（源码 120-134 行）
req_url = f'http://{worker_ip}:{PROJ_CONFIG.worker_port}/modification/container/'
req_data = {'user': user, 'topo': topo, 'name': name}
result = requests.post(req_url, json=req_data)   # 转发到 Worker
if result.json()['code']:
    return {'code': 1, 'msg': '节点创建成功'}
else:
    data_handler.rollback_db()  # Worker失败，回滚Redis
```

### 第二步：Worker API — `webserver/api/dynamic_modify/worker_container_api.py`

Worker 收到请求后，`DynamicContainerAPI.post()` 核心逻辑（源码第 55-76 行）：

1. **解析请求** — 取出 `user`、`topo`、`name`
2. **获取 Redis 数据** — 连接到该用户对应的 Redis DB，拿到节点的完整元数据
3. **创建实体** — 实例化 `DynamicNeCreator(user, topo, name, re_cli)`，调用 `create_and_run()`

```python
# 源码 68-76 行
user_db_map = UserMapRedis()
re_cli = user_db_map.get_user_db(user)
user_db_map.close()
ne_creator = DynamicNeCreator(user, topo, name, re_cli)
ne_creator.create_and_run()    # 真正创建 Docker 容器
ne_creator.close()
return {'code': 1, 'msg': '创建节点成功'}
```

### 第三步：底层 — `Service_layer/NEManager.py` 里的 `DynamicNeCreator`

`create_and_run()` 不在 API 文件里，它属于服务层，最终会调用 `Implement_layer` 的 Docker 命令启动一个容器。

## Klonet 一句话

**Klonet（也叫 VEMU）是一个网络模拟平台。** 你写 Python 脚本告诉它"我要两台主机 + 一台交换机，连起来"，它就在后台用 Docker 帮你把整套网络跑起来，你可以在节点上执行命令、配链路带宽，做完实验就删掉。

## 怎么用？三步走

### 第一步：设计拓扑（搭积木）

```python
from vemu_api import *

# 连上平台
project_manager = ProjectManager("你的用户名")

# 选镜像
images = image_manager.get_images()   # 看看有哪些系统可用
ubuntu = images["ubuntu"]

# 搭拓扑：两台主机中间连一台交换机
topo = Topo()
h1 = topo.add_node(ubuntu, node_name="h1")
h2 = topo.add_node(ubuntu, node_name="h2")
s1 = topo.add_node(images["ovs"], node_name="s1")
topo.add_link(h1, s1, src_IP="192.168.1.1/24")
topo.add_link(s1, h2, dst_IP="192.168.1.2/24")
```

### 第二步：部署，平台帮你跑起来

```python
project_manager.deploy("我的实验", topo)
# 等一会儿，拓扑就变成真实运行的 Docker 容器了
```

### 第三步：做实验、用完删掉

```python
# 在节点上执行命令
cmd_manager.exec_cmds_in_nodes({"h1": ["ping -c 3 192.168.1.2"]})

# 动态加节点 / 删节点
node_manager.dynamic_add_node("h3", ubuntu)

# 做完实验，删除整个项目
project_manager.destroy("我的实验")
```
