# 项目需求说明

## 背景问题

同学们现在会广泛使用 ChatGPT、Copilot、Claude Code、Codex 等通用 AI 工具辅助编程，但在 Klonet 项目中会带来几个问题。

### 过程黑盒化

同学使用 AI 时，经常只留下最终代码，缺少过程记录：

- 需求是怎么理解的。
- 中间尝试了哪些方案。
- 遇到过什么报错。
- Bug 是怎么修掉的。
- 哪些功能和预期还有差异。

这会导致老师或项目负责人只能“看结果”，很难“看过程”。

### 代码风格碎片化

不同同学使用不同 AI 工具，生成的代码风格不同：

- 命名方式不统一。
- 注释风格不统一。
- 模块边界不统一。
- 测试方式不统一。
- 对 Klonet 框架规范理解不一致。

这会增加后续 review、合并和维护成本。

### 知识传承断层

Klonet 的开发和维护经验可能分散在：

- 师兄师姐的脑子里。
- 零散聊天记录里。
- 历史代码里。
- 没有系统整理的踩坑经验里。

新同学上手时需要反复问人，核心成员毕业或离开后，经验容易流失。

## 项目目标

本项目要开发一个 Klonet 专用教学协作 Agent，承担三个角色：

1. 编程助手
2. 数字导师
3. 项目过程记录与总结工具

具体目标：

- 帮同学学习 Klonet。
- 帮同学按规范开发 Klonet 功能。
- 帮老师查看项目进度、开发过程和验收差异。
- 把 Klonet 经验沉淀为可检索的团队知识资产。

## 功能需求

### 1. 项目管理

每个同学、每个项目都应该有独立 Markdown 项目日志。

文件路径建议：

```text
journals/{user_id}/{project_id}.md
```

项目日志需要记录：

- 项目目标
- 当前状态
- 需求与预期功能
- 开发计划
- 执行记录
- 遇到的问题
- 测试与验证
- 功能差异与验收建议
- 下一步

价值：

- 同学可以复盘开发过程。
- 老师可以查看真实进度。
- Agent 可以读取项目日志恢复上下文。
- 历史项目日志可以进入知识库，帮助后来的同学。

### 2. Klonet 知识库

知识库要覆盖：

- Klonet 架构设计。
- Klonet 开发规范。
- Klonet 源码说明。
- Klonet 维护经验。
- 常见报错和踩坑记录。
- 历史项目日志。
- 代码风格指南。

Mentor Agent 回答 Klonet 问题前应该优先检索知识库。

Coding Agent 写代码前应该检索：

- 相关规范。
- 相似实现。
- 历史踩坑。
- 当前项目日志。

### 3. 代码规范

Coding Agent 生成代码时要尽量和当前项目风格保持一致。

要求：

- 中文注释清楚解释目的。
- 模块职责明确。
- 类封装但不过度抽象。
- 优先沿用已有项目模式。
- 不为了炫技引入复杂设计。
- 修改代码后说明影响范围和验证结果。

### 4. 项目开发

Coding Agent 不应该只是生成代码，而应该形成完整开发闭环：

```text
理解需求
-> 制定计划
-> 检索 Klonet 知识
-> 读取相关文件
-> 修改代码
-> 运行测试
-> 查看 diff
-> 写项目日志
-> Review / Reflection
```

### 5. Token 与速度

项目范围固定在 Klonet，因此可以做针对性优化：

- 预先索引 Klonet 知识。
- 常见任务模板化。
- 只注入当前任务相关上下文。
- 工具输出过长时只给摘要。
- 项目日志定期压缩。

后续要通过实验验证：

- token 是否更少。
- 完成时间是否更短。
- 工具调用次数是否减少。
- 测试通过率是否提高。
- 人工 review 问题是否减少。

## 非功能需求

### 安全性

- 不泄露 `.env`、API Key、token、密码等敏感信息。
- 文件读写限制在当前 workspace 内。
- 危险命令需要拒绝或人工确认。
- 不允许直接 push 到主分支。

### 可维护性

- 模块边界清晰。
- 代码注释适合学习。
- Prompt 分层管理。
- Agent Profile 控制行为差异。
- 工具执行统一经过 ToolExecutor。

### 可评估性

需要准备 eval cases，用于后续对比实验：

- Mentor 问答 case。
- Coding 开发 case。
- Error 报错排查 case。

每个 case 需要包含：

- 输入问题或任务。
- 期望行为。
- 是否需要改代码。
- 验收标准。
