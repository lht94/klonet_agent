# Mentor Agent 优化计划

## 目标

Mentor Agent 已具备 Klonet 知识检索、项目日志读取、问题范围路由和基础问答能力。
后续演进必须按以下顺序进行：

1. 阶段一：优化回答质量。
2. 阶段二：增加只读环境感知。
3. 阶段三：增加受控环境操作。

阶段一验收通过前，不实施环境感知；阶段二稳定前，不开放环境修改能力。

## 阶段一：优化回答质量

### 阶段目标

- 回答直接覆盖用户问题，不绕到无关内容。
- 简单问题保持简洁，复杂问题按需要展开。
- Klonet 关键事实有证据，推断和不确定内容有明确标记。
- 相似问题的回答结构和质量保持稳定。
- 优化质量时不显著增加 token、耗时和模型调用次数。

### 1. 建立真实 Mentor 评测集

从真实使用记录中整理 15 至 30 个问题，覆盖：

- Klonet 架构与核心概念。
- 环境安装与平台部署。
- 启动、停止和运行状态。
- 拓扑部署与关键流程。
- 常见报错和故障排查。
- 源码入口与调用链定位。
- 项目进度和验收差异。
- 通用技术和 mixed 问题。

每个 case 记录输入、任务类型、必须包含、禁止出现、期望来源、是否允许推断、
建议长度和是否需要检索。

评测指标包括正确性、直接性、需求覆盖率、证据可靠性、无关内容比例、回答长度、
token 和总耗时。

### 2. 优化问题理解

Mentor 回答前先识别概念解释、故障排查、源码定位、部署指导、项目进度、
通用技术或 mixed 问题。

必须保留原始输入中的否定条件、前置条件和明确需求，不允许在查询改写或工具调用后
改变问题方向。例如“需要 Rust 编译环境”不能被改写成“使用 Rust 开发 Docker 控制器”。

### 3. 优化回答结构

通用规则：

1. 第一段直接给结论。
2. 只解释理解结论所必需的原因。
3. 不重复用户问题，不汇报内部检索过程。
4. 不机械追加学习建议、源码路径或下一步。
5. 没有证据时说明不确定，不生成 Klonet 架构推测。

任务结构：

- 概念问题：结论、核心原理、必要的相关模块。
- 故障排查：最可能原因、排查顺序、判断依据。
- 源码解释：入口、调用链、关键状态、注意点。
- 部署指导：推荐方案、当前前提、执行步骤、验证方式。
- 项目进度：当前状态、已完成、阻塞项、下一步。

建议长度：

- 简单事实：100 至 300 字。
- 概念解释：300 至 600 字。
- 故障排查：500 至 1000 字。
- 源码和复杂流程：按必要步骤展开。

### 4. 优化证据组织

- 区分知识文档、源码事实、项目日志和模型推断。
- Klonet 关键结论尽量提供来源路径。
- 去除重复、过时和冲突证据。
- weak 结果不能作为确定事实，none 结果必须说明资料缺口。
- generic 问题以通用知识为主，Klonet RAG 只能作为 secondary。
- mixed 问题分别组织通用知识和 Klonet 证据。

### 5. 优化检索方式

先根据真实评测定位现有 BM25 的误检和漏检，再逐步形成：

精确匹配 + BM25 + metadata 分层权重 + 向量召回 + 少量候选 reranker。

向量检索采用 OpenAI 兼容 embedding API：

- 通过 base_url、model 和 API key 配置。
- 只发送 public 或已脱敏的知识 chunk。
- 向量索引作为本地生成产物，不提交 Git。
- API 不可用时自动退回 BM25。
- API 路由、类名、函数名和文件路径的精确匹配仍保持最高优先级。

是否引入向量检索，以真实 eval 是否存在语义漏检为依据。

### 6. 控制上下文与性能

- 限制检索次数、top-k、工具结果和证据长度。
- 去除重复 Prompt、无关历史和重复证据。
- 普通问答不增加独立 Reflection 模型调用。
- 轻量回答检查尽量在同一次生成中完成。
- Trace 记录每轮 token、耗时、检索次数和工具调用次数。

### 阶段一验收

- Mentor 直接回答用户核心问题。
- 简单问题不再过度展开。
- 复杂问题不遗漏用户明确要求。
- Klonet 事实有证据，推断有标记。
- 不主动生成无关 Klonet 内容。
- 真实评测中的质量稳定提升，token 和耗时不明显增加。

## 阶段二：增加只读环境感知

### 阶段目标

Mentor 根据 Agent 所在服务器的真实状态调整回答，但不能修改环境。
第一版只感知本机，不通过 SSH 检查其他服务器。

### 感知范围

- Ubuntu 版本、内核、架构、CPU、内存、磁盘和虚拟化支持。
- Docker、Compose、镜像、容器和网络。
- Python、Rust、OVS、KVM 和 libvirt。
- Redis、MySQL、RabbitMQ、Nginx。
- 当前端口、服务和 Klonet 进程状态。
- 允许目录中的源码、Compose、Dockerfile 和部署配置。

### 安全边界

- 环境检查工具默认只读。
- 文件读取限制在源码目录、部署目录和系统配置白名单。
- 不读取 .env、私钥、token、密码和凭据正文。
- 配置内容进入回答前必须脱敏。
- 检查失败标记为“未检查”，不能当作“未安装”。

### 回答适配与验收

Mentor 区分“已检测存在、检测缺失、未检查”。已有镜像、依赖或配置时，
跳过对应标准安装步骤，只说明剩余差异和验证方式。

阶段验收要求同一问题能根据不同环境给出不同建议，所有工具保持只读，
且不泄露敏感配置。

### Ops 后续优化计划表

Ops 模式不应继续依赖 Mentor 的通用 `task_type` 分类作为主控逻辑。
`concept`、`code_lookup`、`development` 和 `project_progress` 这类教学/源码/项目进度标签
容易把运行态诊断带偏，也不应决定工具是否可用或安全边界是否成立。

Ops 的工具权限由 `ops` profile 固定决定；安全边界由工具白名单、执行器护栏、
OperationPlan、确认协议和 recipe allowlist 决定。Ops 的“识别”只作为辅助路由和槽位提取，
用于决定优先检查什么、如何组织证据和是否进入受控计划流程。

| 优先级 | 优化项 | 当前问题 | 目标设计 | 验收标准 |
| --- | --- | --- | --- | --- |
| P0 | Ops 路由从通用意图分类中解耦 | 端口、screen、日志和路径混合问题可能被识别成 `code_lookup`、`concept` 或 `project_progress`，导致诊断方向跑偏 | 新增 Ops 专用轻量路由/slot extractor，只提取 `port`、`path`、`platform`、`component`、`action`、`risk` 等运维槽位；不再把通用 `task_type` 作为 Ops 主控 | Ops 不再显示 `已识别：concept/code_lookup/...`，而显示“目标/线索/模式”，例如 `目标：端口占用诊断；线索：port=5045, component=web_terminal_main.py；模式：只读诊断` |
| P0 | 工具可用性不依赖识别结果 | 如果识别错，环境工具可能不可用或模型不敢调用，导致问题回答不上来 | Ops 模式始终暴露只读环境工具、知识检索、配置读取、screen 检查和 OperationPlan 工具；工具权限只由 profile 控制 | 同一句运维问题即使路由槽位不完整，也仍可调用 `inspect_klonet_runtime`、`inspect_screen_session`、`read_ops_file` 等工具继续排查 |
| P0 | 安全边界不依赖识别结果 | 仅靠识别成 `unsafe_action` 或 prompt 拒绝不可靠；识别错也不能导致危险工具可用 | `run_command`、任意 kill、任意 systemctl、任意文件删除不进入 Ops 工具集；修改环境只能通过 OperationPlan、用户确认和已验证 recipe | 即使模型误判或用户诱导，未经 `confirm <plan_id>` / `confirm-step <plan_id> <step_id>` 的修改操作也会被执行器拒绝 |
| P1 | Ops 输出结构由运维目标决定 | 通用回答策略会把运行态诊断写成概念解释、源码解释或项目进度 | 按 Ops 目标选择固定输出结构：已确认、未确认、证据来源、风险、下一步只读检查或计划确认命令 | “5045 被占用”这类问题必须输出 PID/cmd/cwd 的确认状态；证据不足时明确列出缺口，而不是给概念解释 |
| P1 | 写操作请求进入计划流程 | 用户说“重启/部署/销毁/kill”时，当前容易混入 Coding 或普通建议 | Ops 识别到 `action=restart/deploy/destroy/stop/kill` 时，只能生成 OperationPlan 或拒绝直接执行；Coding 模式不处理服务器运行环境修改 | Mentor 建议切 Ops；Ops 生成计划；Coding 不被推荐用于服务器 kill、重启或销毁 |

## 阶段三：增加受控环境操作

### 阶段目标

Mentor 在用户授权后可以部署 Klonet 或准备 Klonet 环境，同时保持可审查、
可停止、可验证和有限回滚。

### 执行流程

环境检查 -> 检索部署证据 -> 生成 DeploymentPlan -> 用户确认计划 ->
普通步骤批量执行 -> 特权步骤逐次确认 -> 验证 -> Journal/Trace ->
必要时回滚本次新增资源。

授权命令：

- confirm plan_id：批准普通步骤批量执行。
- confirm-step plan_id step_id：批准单个特权步骤。
- abort plan_id：停止后续执行。
- 模型不能为自己授权。

sudo、覆盖配置、修改端口、防火墙、Docker daemon、共享服务重启和已有资源清理
必须二次确认。

### 结构化部署执行器

不向 Mentor 直接开放任意 run_command。部署使用版本化 DeploymentRecipe，覆盖：

- 系统预检。
- Git 拉取并锁定 commit。
- 依赖安装。
- Docker、OVS、KVM 和基础服务准备。
- 配置渲染与数据库初始化。
- Master、Celery、Terminal、Worker 和 Nginx 启动。
- 健康检查与最小拓扑验收。

只有当前源码版本存在已验证 recipe 时，才允许完整自动部署。

### 主机、凭据与失败策略

- 支持专用服务器和共享服务器，共享环境优先复用健康依赖。
- Ubuntu 版本不固定，未知版本先做兼容性预检。
- 源码经批准后从 Git 拉取并锁定 commit。
- 凭据只通过环境变量名或权限文件路径引用，不进入 Prompt、回答、Journal 或 Trace。
- 任一步失败后停止后续步骤。
- 只自动清理本计划新建的 checkout、容器、网络、临时文件和服务单元。
- 不自动修改或删除部署前已有配置和共享服务。

### 阶段三验收

- 用户在执行前能看到完整计划，未确认计划无法执行。
- 模型不能绕过二次确认。
- 每步有明确输入、输出、验证和失败原因。
- 不破坏服务器已有服务和资源。
- 完成后通过依赖、进程、端口、Worker 注册和最小拓扑检查。
- Trace 能还原环境、计划、授权、执行、验证和回滚全过程。

## 最终目标

Mentor Agent 最终具备三层能力：

1. 基于可靠证据，给出准确、清楚、简洁的教学回答。
2. 感知本机环境，给出符合当前状态的针对性建议。
3. 经用户授权，安全执行 Klonet 部署与环境准备。

当前唯一实施重点是阶段一。阶段二和阶段三只保留目标与边界，
在前一阶段验收完成前不提前实现。
