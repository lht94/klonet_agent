# Klonet 专用教学协作 Agent 项目总览

## 项目定位

本项目要做的不是一个通用聊天机器人，也不是简单复刻 Claude Code，而是一个面向 Klonet 项目的**专用教学协作 Agent**。

它的目标是服务教研室或课程团队中的同学，帮助大家学习、开发、维护 Klonet，同时帮助老师或项目负责人看到开发过程、进度差异和验收风险。

## 两个核心 Agent

项目最终包含两个 Agent，但它们不应该使用两套完全独立的框架，而应该是：

> 一个共享底层框架 + 两个不同 Agent Profile。

### Klonet Mentor Agent

定位：Klonet 数字导师。

主要职责：

- 回答 Klonet 架构、设计、源码、维护相关问题。
- 帮同学理解历史代码和模块职责。
- 排查 Klonet 常见报错。
- 根据项目日志理解某个同学当前开发进度。
- 把高价值问答沉淀为知识库候选内容。

推荐范式：

```text
RAG + ReAct + 轻量 Reflection
```

原因：

- Mentor Agent 的关键是“基于证据解释”，所以必须优先检索 Klonet 知识库。
- ReAct 适合边检索、边读源码、边修正判断。
- Reflection 只在高风险或证据不足时轻量使用，避免每次问答都增加成本。

### Klonet Coding Agent

定位：Klonet 编程协作助手。

主要职责：

- 帮同学按 Klonet 规范实现新功能。
- 读取已有代码，找到相似实现。
- 修改 workspace 中的代码。
- 按 Klonet 的方式运行测试和验证。
- 生成 diff、总结改动、记录项目日志。
- 做轻量 review，检查功能、测试、规范和安全风险。

推荐范式：

```text
Plan-and-Solve + 工具循环（类 ReAct）+ Review / Reflection
```

原因：

- 写代码需要先计划，再执行，再验证。
- 工具循环负责读文件、写文件、跑测试、查看 diff。
- Review/Reflection 负责最后的质量检查，尤其是代码规范和验收差异。

## 核心差异化价值

相比 Claude Code、Codex 等通用 coding agent，本项目的价值不在于“模型更强”，而在于目标函数不同。

### 1. 项目管理

每个用户、每个项目都有自己的 Markdown 状态机：

```text
journals/{user_id}/{project_id}.md
```

它记录：

- 项目目标
- 当前状态
- 需求与预期功能
- 开发计划
- 执行记录
- 遇到的问题
- 测试与验证
- 功能差异与验收建议
- 下一步

这样同学可以复盘自己的开发过程，老师也可以查看项目进度和卡点。

### 2. Klonet 专属知识库

知识库沉淀 Klonet 的：

- 架构设计
- 开发规范
- 源码解释
- 历史踩坑
- 报错处理
- 项目日志
- 验收经验

通用 Agent 每次都要重新理解项目，而 Klonet Agent 可以逐步积累组织内部的长期知识。

### 3. 代码规范统一

Coding Agent 生成的代码要遵循 Klonet 项目风格，包括：

- 目录结构
- 命名规范
- 注释风格
- 测试方式
- 常见禁止写法
- 已有模块边界

这可以减少不同同学使用不同 AI 工具带来的代码风格碎片化问题。

### 4. Klonet 项目开发流程更熟悉

通用 coding agent 可以写代码，但不一定知道 Klonet 的功能应该怎么验收、怎么测试、怎么和已有模块对齐。

Klonet Coding Agent 应该知道：

- 新功能该看哪些模块。
- 该参考哪些相似实现。
- 该运行哪些测试。
- 该如何记录验收差异。
- 该如何提醒同学下一步做什么。

### 5. Token 和速度潜力

因为项目范围固定，Klonet Agent 可以：

- 预索引 Klonet 知识。
- 缓存常用规范。
- 只注入当前任务相关上下文。
- 使用模板化开发流程。
- 避免每次从零理解项目。

因此它理论上比通用 Agent 更省 token、更快完成常见任务。这个优势后续需要通过 evals 做对比实验验证。

## 当前技术路线

第一阶段：

- 不重度使用 LangGraph。
- 不直接上复杂向量数据库。
- 使用自研轻量工具循环框架。
- 使用本地 JSONL 知识索引。
- 使用 Markdown/JSONL 保存记忆和项目日志。
- 目标是本地 CLI 可用，方便学习和毕业设计展示。

第二阶段：

- 引入 SQLite 管理多用户、多项目元数据。
- 知识库升级为 BM25 + 向量检索。
- 实现独立 ReviewAgent。
- 加入更完整的 trace 和 eval。

第三阶段：

- 如果需要生产化，再考虑 LangGraph、Web API、权限系统、持久化任务流和人工审批。
