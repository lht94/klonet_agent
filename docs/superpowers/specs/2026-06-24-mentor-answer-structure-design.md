# Mentor 回答结构优化设计

## 背景与目标

Mentor 已具备问题路由、Klonet 知识检索和基础问答能力，但当前回答结构主要依赖通用 Prompt，容易出现结论靠后、简单问题过度展开、机械追加下一步，以及不同任务类型结构不稳定的问题。

本次优化只调整 Mentor 的回答组织方式，不修改检索算法，不增加模型调用，也不实施环境感知或环境操作。

目标如下：

- 第一段直接回答用户的核心问题。
- 只保留理解结论所需的信息。
- 根据任务类型使用稳定、合适的回答结构。
- 简单问题保持简洁，复杂问题按需展开。
- Klonet 事实缺少可靠证据时明确说明不确定，不生成架构推测。
- 不机械追加学习建议、源码路径或下一步。
- 不增加模型调用次数和明显的上下文开销。

## 方案选择

采用“静态 Prompt 通用规则 + 编排器按任务类型注入本轮回答策略”的方案。

没有选择仅修改 `MENTOR_PROMPT`，因为所有任务模板同时存在于静态 Prompt 中时，模型仍需自行选择结构，稳定性有限。也不采用结构化输出协议或回答后处理，因为这会增加实现复杂度，并可能破坏自然语言回答或引入额外模型调用。

## 组件设计

### 通用 Prompt

`MENTOR_PROMPT` 只保留跨任务通用的回答规则：

1. 第一段直接给出结论。
2. 只解释理解结论所必需的原因。
3. 不重复用户问题，不汇报内部检索过程。
4. 不机械追加学习建议、源码路径或下一步。
5. 没有可靠证据时说明不确定，不生成 Klonet 架构推测。

删除或改写“尽量包含下一步学习或排查建议”等与目标冲突的要求。Mentor 的默认 workflow 由 `retrieve -> explain -> suggest next step` 调整为不强制追加下一步的表达。

### 回答策略模块

新增独立的轻量回答策略模块。该模块接收现有路由结果中的 `task_type` 和原始用户输入，返回一段只用于当前轮的系统策略文本。

策略映射如下：

- `concept`：结论、核心原理、必要的相关模块。简单事实建议 100 至 300 字；需要解释原理时建议 300 至 600 字。
- `troubleshooting`：最可能原因、排查顺序、判断依据，建议 500 至 1000 字。
- `code_lookup`：入口、调用链、关键状态、注意点，按解释调用链所需步骤展开。
- `development`：当原始输入包含部署、安装、启动、配置等部署意图时，使用推荐方案、当前前提、执行步骤、验证方式；其他开发问题沿用紧凑的开发指导结构。
- `project_progress`：当前状态、已完成、阻塞项、下一步。这里的“下一步”属于任务结构本身，不受“不机械追加下一步”规则限制。
- `general`：直接回答通用技术问题，不主动加入 Klonet 模块、源码路径或学习建议；长度按问题复杂度控制。

第一版复用现有 Router，不新增任务分类模型。简单事实和部署意图使用小范围、可测试的关键词判断；无法稳定判断时采用更宽松的概念或开发结构，不强制伪装成精确分类。

### 编排器集成

`AgentOrchestrator.single_chat()` 在完成问题路由后构建两条当前轮系统消息：

1. 现有的“本轮问题范围”，负责 scope、检索边界和原始输入保护。
2. 新增的“本轮回答策略”，负责回答结构、长度和证据表达。

两条消息只在当前工具循环内存在。最终回答生成后，从 history 中移除，避免污染下一轮或被写入长期上下文。

回答策略只提供生成约束：

- 不新增 Reflection 或 Judge 调用。
- 不在生成后按字符数硬截断。
- 不强制添加固定 Markdown 标题。
- 不改变工具权限、检索预算和已有路由结果。

## 数据流

```text
原始用户输入
  -> QueryRouter 生成 scope/task_type
  -> 构建本轮问题范围消息
  -> AnswerPolicy 构建本轮回答策略消息
  -> LLM/工具循环
  -> 按策略生成最终回答
  -> 移除两条本轮临时消息
```

## 异常与降级

- 未知或不受支持的 `task_type` 使用通用回答策略，不能阻断对话。
- 无可靠 Klonet 证据时，策略要求明确说明资料不足；不能用模型推断补齐架构事实。
- 问题同时具有多个意图时，以 Router 的主任务类型为主体；mixed 仍由现有 scope 规则负责区分通用知识与 Klonet 证据。
- 长度是建议范围，不作为硬性截断条件，避免切断代码、步骤或关键说明。

## 测试设计

采用 TDD，先补失败测试，再实现最小代码。

测试至少覆盖：

- `MENTOR_PROMPT` 包含五条通用结构规则，并且不再强制追加下一步。
- Mentor 默认 workflow 不包含 `suggest next step`。
- 每个现有 `task_type` 生成正确的结构提示。
- 部署意图在 `development` 下获得部署指导结构。
- 简单事实获得 100 至 300 字建议，普通概念解释获得 300 至 600 字建议。
- 未知任务类型安全降级为通用策略。
- 编排器首轮调用中包含“本轮回答策略”。
- 完成对话后，回答策略消息不会残留在返回的 history 中。
- 原有 scope、检索预算和 Prompt 风格测试继续通过。

## 验收标准

- Mentor 的首段回答约束明确存在。
- 概念、故障排查、源码解释、部署指导和项目进度具备各自结构。
- 简单事实和复杂问题具有不同长度建议。
- 不再默认附加学习建议、源码路径或下一步。
- weak/none 证据不能被表达为确定的 Klonet 事实。
- 不增加模型调用次数，不新增外部依赖。
- 全部相关自动化测试通过。

## 范围外事项

本次不实现以下内容：

- 真实 Mentor Answer Eval runner。
- 向量检索或 reranker。
- 回答生成后的 LLM Judge。
- 强制 JSON/Markdown 输出协议。
- 环境感知和环境操作。
