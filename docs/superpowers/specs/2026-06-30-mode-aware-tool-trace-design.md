# Mentor 与 Ops 模式输出轨迹设计

## 背景

当前 `AgentOrchestrator` 通过同一个进度输出开关同时服务 Mentor 和 Ops。
因此 Mentor 在最终“思考摘要”之外，还会逐次打印“正在调用工具”和“工具完成”；
Ops 则在每次工具调用中重复输出调用、完成、单行结果摘要和固定下一步。

这些输出与两个模式的职责不匹配：Mentor 应提供教学式聚合摘要，Ops 应提供可审计、
可验证的实时执行轨迹。

## 目标

- Mentor 保留“正在理解你的问题”“已识别”“正在组织回答”等阶段进度。
- Mentor 隐藏逐次工具调用轨迹，最终继续输出聚合后的“思考摘要”和回答。
- Ops 使用类似 Coding Agent 的“动作 → 观察”轨迹展示真实工具执行过程。
- Ops 不再重复输出“正在调用工具”和“工具完成”。
- Ops 不再输出内容固定、没有实际决策信息的“下一步”。
- `brief` 模式继续只输出最终回答。
- 展示内容不得包含模型隐藏推理或未脱敏的敏感参数。

## 非目标

- 不改变工具选择、工具循环、权限边界或最大调用轮数。
- 不增加独立模型调用来生成过程说明。
- 不重写 Mentor 的思考摘要结构。
- 不将完整工具返回值原样打印到终端。

## 方案

采用确定性的“动作 → 观察”输出。

每次工具执行前，编排器根据工具名和关键参数输出动作。例如：

```text
Klonet Agent：正在搜索源码：vemu_frontend
```

工具返回后，编排器从真实结果中提取最多三条有效信息并输出观察。例如：

```text
Klonet Agent：观察：找到 2 条相关结果
  - vemu_frontend 是平台 Web 前端
  - 关联源码目录：frontend/...
```

如果下一轮模型决定继续调用工具，新的动作行本身就是实际下一步；如果模型直接回答，
则不产生虚构的下一步说明。

## 组件设计

### 模式可见性

将阶段进度和工具轨迹拆成两个独立判断：

- 阶段进度：Mentor 与 Ops 在非 `brief` 模式下可见。
- 工具轨迹：仅 Ops 在非 `brief` 模式下可见。
- 思考摘要：仅非 Ops 且非 `brief` 模式下可见，保持现有行为。

工具执行逻辑只调用一次 Ops 轨迹入口；Mentor 仍收集 `tool_events`，供最终思考摘要使用，
但不实时打印这些事件。

### 动作格式化

动作格式化使用确定性工具映射，不让模型生成描述。常用工具使用面向用户的动词：

- `search_knowledge`：正在检索知识库。
- `search_code`：正在搜索源码。
- `list_source_files` / `list_files`：正在查看目录。
- `read_source_file` / `read_file` / `read_ops_file`：正在读取文件。
- 环境、日志和 screen 检查工具：正在检查对应运行状态。

动作只展示白名单关键参数，如 `query`、`path`、`session_name`。字符串统一压缩空白并限制长度；
未知工具退回“正在执行工具：<tool_name>”，但不打印完整参数对象。

### 观察摘要

观察摘要直接基于工具真实返回值生成，不调用模型：

1. 去除空行。
2. 保留能表达命中数量、文件路径、状态、错误或结论的有效行。
3. 最多展示三条，限制单行和总长度。
4. 多余内容用“其余内容已省略”提示。
5. 空结果显示“工具未返回可展示结果”。
6. `Error:`、失败、未找到目录等结果使用明确的失败或未命中措辞，不伪装为成功。

首条较短结果可以与“观察”同行；多条结果使用缩进列表。这样既能看到真实摘要，
又避免把大段源码、日志或知识库全文倾倒到终端。

### 数据流

```text
模型发起 tool_call
  → Ops 输出格式化动作（Mentor 静默）
  → 执行现有 use_tool
  → Ops 输出真实观察摘要（Mentor 静默）
  → 记录 tool_event 并回填模型上下文
  → 模型决定下一次工具调用或最终回答
```

终端输出与模型上下文继续分离，轨迹文本不会被追加到对话历史。

## 错误与安全处理

- 参数解析和工具执行行为保持原样；本改动只负责展示。
- 动作参数采用白名单，不显示 token、密码、环境变量正文或任意完整参数。
- 工具结果沿用现有执行层的脱敏结果，并在展示层做长度限制。
- 格式化器自身遇到异常时使用安全兜底文案，不影响工具结果进入模型上下文。

## 测试设计

使用现有 `tests/test_orchestrator_controls.py` 做行为回归：

1. Mentor 保留理解、识别和组织回答阶段。
2. Mentor 不出现逐次工具动作、完成或观察输出。
3. Mentor 最终仍出现思考摘要，并包含聚合后的工具动作与依据。
4. Ops 出现带关键参数的动作和来自真实结果的观察。
5. Ops 不出现“工具完成”“工具结果摘要”“下一步”旧文案。
6. 多行结果最多显示三条，并明确标记省略。
7. 空结果和错误结果使用明确、安全的兜底文案。
8. `brief` 模式不输出阶段进度、工具轨迹或思考摘要。

先增加失败测试，再实现最小格式化和模式分流，最后运行编排器相关测试与完整测试集。

## 验收示例

Mentor：

```text
Klonet Agent：正在理解你的问题...
Klonet Agent：已识别：concept
Klonet Agent：正在组织回答...
Klonet Agent：思考摘要：
...
Klonet Agent：<最终回答>
```

Ops：

```text
Klonet Agent：已识别：concept
Klonet Agent：正在组织回答...
Klonet Agent：正在检索知识库：vemu_frontend
Klonet Agent：观察：找到 2 条可靠证据
  - vemu_frontend 是平台 Web 前端
  - 来源：knowledge/...
Klonet Agent：正在搜索源码：vemu_frontend
Klonet Agent：观察：未在源码中找到匹配内容。
Klonet Agent：<最终回答>
```
